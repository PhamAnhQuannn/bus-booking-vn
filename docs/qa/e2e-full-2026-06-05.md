## Executive summary

**The operator↔traveler business loop works end-to-end** — an operator-released trip is discoverable in customer search, a traveler can hold + pay it, the paid booking surfaces on the operator manifest, seat availability decrements across the boundary, the operator can check the passenger in, and an operator trip-cancel refunds the traveler (booking → `trip_cancelled` + `refund_out` ledger). The visibility/approval gate works both directions.

**One blocking bug breaks the traveler payment UI** (`P1`): the dev stub-pay gateway buttons 500 on every click, so no traveler can complete a payment through the browser. The cross-persona run proves the rest of the money loop by completing payment through the same `processPaymentWebhook` the real gateway uses (a faithful bypass) — but in the live UI, payment is currently dead. Prior smoke runs hid this because they ran skip-pay.

## 🔴 Prioritized issues

### P1 — Stub payment UI button returns HTTP 500 (traveler cannot pay) — BLOCKER
- **Where:** `/dev/stub-pay` → "Thanh toán" (success) and "Thất bại" (fail) buttons; `app/dev/stub-pay/actions.ts:43`, form in `app/dev/stub-pay/page.tsx:80-97`.
- **Symptom:** POST `/dev/stub-pay` → 500; page stuck on the gateway; `Error: stub-pay: invalid outcome ` (empty `outcome`). Server trace shows the action invoked as `submitStubPayment({})` — empty FormData.
- **Root cause:** the two submit buttons carry `name="outcome" value="success|fail"` + `formAction={submitStubPayment}`. Under Next 16.2.6 / React 19 server actions, the **submitter button's `name`/`value` is not delivered** to the action's FormData, so `formData.get('outcome')` is `''` → the `outcome !== 'success' && outcome !== 'fail'` guard throws.
- **Impact:** every booking dies at the pay step — `/booking/review → /dev/stub-pay → ✗`. This is the only Phase-1 payment path (`PAYMENTS_STUB=true`), so checkout is fully broken in the browser. (Backend `processPaymentWebhook` is healthy — completing payment via the webhook works, see C2/C6.)
- **Fix options (dev-only file, pick one):**
  1. Bind the outcome per button: `formAction={submitStubPayment.bind(null, 'success')}` / `.bind(null, 'fail')` and drop the `name`/`value` reliance.
  2. Two separate `<form>`s, each with a hidden `<input name="outcome" value="success|fail">`.
  3. Two thin wrapper server actions (`paySuccess` / `payFail`) that call a shared impl.
- **Evidence:** traveler run BROKEN×5, cross-persona BROKEN×4, server log `app\dev\stub-pay\actions.ts:43`.

### P2 — Operator profile save returns HTTP 400 — degraded
- **Where:** `/op/profile` "Lưu hồ sơ"; `PATCH /api/op/profile` → `400 INVALID_PHONE`; client `app/op/(console)/profile/OpProfileClient.tsx`, route `app/api/op/profile/route.ts:101-123`.
- **Root cause:** the client pre-fills `contactPhone` + `notificationPhone` from the loaded profile and re-submits any non-blank field. The **seed** operator's two phones are gitleaks-safe PII masks (`+8490xxxxxx…`, literal `x`) that fail `normalizePhone` → 400. Saving just the display name re-sends the invalid masked phones.
- **Impact:** the seeded operator cannot save their profile until both phone fields are first replaced with valid VN numbers. Real-data operators (valid phones) are unaffected.
- **Fix options:** seed valid + distinct phones for `OperatorUser`; or have the client send only fields the user actually changed (dirty-tracking) so an unchanged masked phone isn't round-tripped; or skip phone normalization when the value is unchanged.
- **Evidence:** operator run WARN×2 (the only operator-side findings).

### P3 — Mobile guest-booking aborted on a click timeout — minor / likely P1-adjacent
- **Where:** traveler mobile (390×844) viewport, `guest-booking` phase: `locator.click: Timeout 30000ms exceeded`.
- **Assessment:** occurs in the pay leg on mobile; most likely the same broken stub-pay button (or a mobile overlay covering it). Re-test after P1 is fixed; if it persists, investigate a mobile-layout element intercepting the click.

**Not a bug (clarified during testing):** the customer form correctly rejects buyer names containing digits ("Họ tên chỉ được chứa chữ cái…") and non-local phone formats. Early cross-persona "stuck" findings were the harness sending a digit-laden name / E.164 phone — fixed in the harness, not the app.
