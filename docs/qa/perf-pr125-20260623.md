## Summary

Removal PR: deletes the entire predefined pickup-area subsystem (4 DB tables, enums,
CRUD API routes, operator UI pages, client libraries) and narrows `PickupKind` from
`station | point | custom` to `station | custom`. Net code reduction of ~2,900 lines.

## Detailed analysis

### New code added (4 files with meaningful additions)

1. **`CustomerForm.tsx`** (+38 lines) -- Replaced `<Select>` dropdown (imported 7 sub-components
   from `@/components/ui/select`) with 2 native HTML `<input type="radio">`. Net bundle
   reduction. Deep import of `validatePickupSelection` from `@/lib/booking/pickupSelection`
   is correct for `'use client'` (avoids barrel server-graph pull per Issue 092b rule).

2. **`lib/booking/pickupSelection.ts`** (+11 lines) -- Pure synchronous validator. No DB calls,
   no imports beyond type definitions. Zero perf surface.

3. **`lib/booking/__tests__/pickupSelection.test.ts`** (+6 lines) -- Unit test assertions only.

4. **Migration SQL** (+101 lines) -- One-time DDL/DML: backfills `point` rows to `custom`,
   drops 4 tables, drops 2 columns from Hold + Booking, narrows the `PickupKind` enum,
   drops the `PickupPlaceKind` enum. Backfill UPDATEs filter on an enum column (`pickupKind`)
   which PG handles efficiently via seq scan (acceptable for a one-time migration on
   early-stage tables). No runtime query impact.

### Indexes

- **Dropped** (via `DROP TABLE`): `@@index([operatorId, isActive])` on OperatorPickupArea,
  `@@unique + @@index` on TripPickupArea, TemplatePickupArea, RoutePickupArea. All belonged
  to the removed tables -- no orphaned index concern.
- **Dropped columns**: `pickupAreaId` and `pickupAreaLabel` on Hold and Booking. These were
  FK columns with implicit Prisma indexes. The FK constraints are explicitly dropped before
  the columns. No remaining query path references these columns.
- **Retained**: All existing indexes on Hold and Booking (including `customPickupRequested`
  index used by manifest filtering) remain intact.

### Schema parity

No new raw-SQL indexes were added, so the Issue 007 rule (raw-SQL indexes expressible in
Prisma DSL must also be declared as `@@index` in `schema.prisma`) is not triggered.

## Verdict

**No performance findings.** This PR is a net positive: it removes 4 tables (and their
associated joins/queries), eliminates several client-side API calls (`pickupAreasClient.ts`
deleted), and reduces the customer booking form's client bundle by removing the Select
component import tree. No new queries, no new API handlers with DB access, no heavy
client imports, no intervals or cron registrations, no caching regressions.
