## Findings

### P1 — Critical (Security/Visibility)

#### 1. Incomplete operator brand masking in test fixture
**File:** `lib/ticketing/__tests__/getTicketVerification.test.ts:49, 76`  
**Issue:** Unit test fixture still hardcodes `'Phuong Trang'` (real Vietnam bus operator). PR masked this real brand to `'Test Bus Co'` in 4 other test files (ticket PDF, customer detail, booking route tests) but missed this ticketing test.  
**Risk:** When repo is public, real company name appears in CI logs / test artifact runs, linking the project to a specific Vietnamese operator.  
**Fix:** Replace both occurrences with `'Test Bus Co'`.

#### 2. .gitignore pattern `~$*` risks capturing unintended files
**File:** `.gitignore:72`  
**Issue:** Pattern `~$*` is intended to match Office temp-lock files (`~$document.docx`) but will match ANY file starting with `~$`, including legitimate version-control artifacts or non-Office files if they follow that naming. On Windows, Office creates files like `~$HO-SO-PHAP-LY-CHI-TIET.doc` so the pattern is correct, but it's overly broad.  
**Risk:** Low immediate risk (Office is the primary source of `~$` prefixes), but brittle. Future developers might create or commit files matching this pattern unaware.  
**Recommendation:** Document or narrow the pattern; e.g., add a comment explaining Office-only intent, or use `~$*.{doc,docx,xlsx}`.

#### 3. Redis password fail-closed only in prod/staging; dev compose unaffected
**File:** `docker-compose.prod.yml:62, 69`, `docker-compose.staging.yml:60, 65`  
**Issue:** PR hardens Redis password to `${REDIS_PASSWORD:?REDIS_PASSWORD required}` in prod/staging but does NOT change `docker-compose.dev.yml`. While intentional (dev runs locally with implicit `changeme`), the inconsistency means a developer copying prod compose to dev without reading the comment could inherit the strict fail-closed behavior.  
**Risk:** Medium — dev setup instructions should be explicit.  
**Note:** Not a blocker; dev compose is local-only and the implicit `changeme` is documented elsewhere. Acceptable asymmetry for public repo.

### P2 — Moderate (Completeness / Maintenance)

#### 4. Brand name replacement incomplete in documentation examples
**File:** `documentation/design-specifications/DS-011-tax-withholding/README.md:284`, `documentation/feature-implementation/FI-006-search-discovery/README.md:113`, `documentation/frontend-design/FD-015-payment-checkout/README.md:22`, `documentation/frontend-design/FD-016-booking-confirmation/README.md:77, 91`, `documentation/frontend-design/FD-017-booking-lifecycle/README.md:60, 108`, `documentation/business/personas/stakeholder-map.md:11, 34`  
**Issue:** 11 references to `Phuong Trang` remain in documentation and design specs. These appear in example JSON payloads, wireframe mockups, and persona/competitive analysis.  
**Context:** These are architectural examples and business context, not executable code. The AGENTS.md mistake log and CLAUDE.md both treat real-company references in documentation differently than in seed/test data — docs are historical/example context, but real names in public examples signal operational partnership.  
**Risk:** Moderate — public visibility of a real operator name in examples implies a specific deployment relationship. Reviewers skimming the repo may assume active partnership or vendor lock-in.  
**Action:** Out of scope for this PR (documentation refactor is separate work); should be tracked as a follow-up issue.

#### 5. devtunnel allowlist now permits broader scope
**File:** `next.config.ts:34`  
**Issue:** PR removed specific tunnel ID `93ppgcdj-3001.usw3.devtunnels.ms` and kept wildcard `*.devtunnels.ms`. Change is correct (removes single-user identifier), but expands the allowed origin scope to any devtunnel subdomain.  
**Risk:** Low — devtunnels are time-limited and single-user sessions. Any attacker would need to compromise VS Code itself or guess an active tunnel URL. Wildcard is acceptable for dev-only mode.  
**Impact:** Intentional security trade-off; wildcard is simpler to maintain than a list of team members' tunnel IDs.
