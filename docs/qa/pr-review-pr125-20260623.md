## Cat 5 -- Rollback path

**Irreversible operations in migration:**

1. `DROP TABLE IF EXISTS "RoutePickupArea"` -- data loss
2. `DROP TABLE IF EXISTS "TripPickupArea"` -- data loss
3. `DROP TABLE IF EXISTS "TemplatePickupArea"` -- data loss
4. `DROP TABLE IF EXISTS "OperatorPickupArea"` -- data loss
5. `ALTER TABLE "Hold" DROP COLUMN IF EXISTS "pickupAreaId"` -- data loss
6. `ALTER TABLE "Hold" DROP COLUMN IF EXISTS "pickupAreaLabel"` -- data loss
7. `ALTER TABLE "Booking" DROP COLUMN IF EXISTS "pickupAreaId"` -- data loss
8. `ALTER TABLE "Booking" DROP COLUMN IF EXISTS "pickupAreaLabel"` -- data loss
9. `DROP TYPE "PickupKind"` (the old one) + recreation without `point` -- enum narrowing
10. `DROP TYPE IF EXISTS "PickupPlaceKind"` -- enum removal
11. `UPDATE` statements converting `point` -> `custom` -- data transformation (non-reversible without backup)

**Data preservation:** The migration includes thoughtful data rescue steps:
- Step 1a: Concatenates `pickupAreaLabel + ' - ' + pickupDetail` for point rows with both
- Step 1b: Copies label into detail for point rows with only a label
- Step 1c: Defensive fallback placeholder for point rows with neither
- Short-label padding to satisfy the CHECK constraint (>=5 chars when custom)
- Sets `customPickupRequested = true` for all migrated point rows

**Rollback documentation in PR:** NONE. The PR body does not mention rollback.

**Verdict: P2 (advisory).** This is a destructive migration with no documented rollback path. However:
- Per CLAUDE.md: "Committed migrations are never edited. Fix via forward migrations."
- The data rescue steps are thorough and defensive
- Pre-prod, this is acceptable; pre-go-live, a DB backup should precede deployment
- A rollback would require a forward migration to recreate tables + restore from backup

Recommendation: Add a brief note in the PR body that this migration is destructive and a DB backup should be taken before running in any environment with real data.
