PRIORITY 1 -- Blocking
================================================================================

  [DOUBLE-DB-HIT-PER-POLL] BankTransferClient.tsx + page.tsx
    path: app/(customer)/booking/bank-transfer/BankTransferClient.tsx:51
    path: app/(customer)/booking/bank-transfer/page.tsx:71

    Each 5s poll cycle fires TWO DB queries, not one:
      (1) Client fetches GET /api/bookings/status?ref=... --> prisma.booking.findUnique
          with select: { status: true }.
      (2) Then router.replace('/booking/bank-transfer?...&r=N+1') triggers an RSC
          re-render of page.tsx, which calls getBookingByRef(bookingRef) -->
          prisma.booking.findUnique with select: { id, confirmationToken, status }.

    So per client per payment flow: 2 queries x 120 polls = 240 DB round-trips.
    At 100 concurrent transfers: 24,000 queries over 10 min.

    The page.tsx server-component query is unnecessary during polling -- it exists
    to validate the bookingRef and pass confirmationToken as a prop to the client
    component. But after the initial page load, confirmationToken never changes --
    it's already in the client component's props.

    Fix options (pick one):
    (a) Preferred: remove the router.replace polling. Use a pure setTimeout loop
        within the same useEffect that only calls fetch(/api/bookings/status).
        Increment refreshCount in local state, not URL params. This eliminates
        the RSC re-render entirely -- 1 query per poll instead of 2.
    (b) If the router.replace pattern is intentional (e.g. for back-button UX),
        move the getBookingByRef call behind a check: skip the DB query when the
        'r' search param is present (the booking was already validated on r=0).

  [DOUBLE-PARSE] webhook/route.ts:49 + processWebhook.ts:89
    path: app/api/payments/bank_transfer/webhook/route.ts:49
    path: lib/payment/processWebhook.ts:89

    verifyWebhook(rawBody) called twice per webhook delivery -- once for the
    no_booking_ref_in_memo early-exit in the route handler, then again inside
    processPaymentWebhook. Each invocation does JSON.parse(rawBody).
    Pure function so no correctness issue, but the SePay webhook payload is parsed
    twice on the happy path.

    Severity: low. JSON.parse on a ~300-byte SePay payload is sub-microsecond.
    The double-parse is a separation-of-concerns tradeoff (the route handler
    doesn't want to couple its pre-check to processPaymentWebhook internals).
    Acceptable as-is. If it ever becomes a bottleneck (it won't), pass the
    pre-verified result into processPaymentWebhook.

  [NO-RATE-LIMIT-ON-STATUS-POLL] api/bookings/status/route.ts
    path: app/api/bookings/status/route.ts
    path: proxy.ts:262

    GET /api/bookings/status is unauthenticated and has no rate limiting.
    proxy.ts line 262: SAFE_METHODS (GET/HEAD/OPTIONS) skip the rate-limit +
    CSRF gate entirely -- they only issue cookies.

    The BOOKING_REF_REGEX validation (line 20) prevents garbage queries, but a
    malicious client can enumerate valid booking refs at unlimited QPS. Each
    valid ref leaks the booking status (awaiting_payment / paid / etc.).

    This is a P2 for launch readiness, not a perf bottleneck per se, but under
    abuse it becomes one: unlimited GET floods hit prisma.booking.findUnique with
    no back-pressure.

    Fix: add a lightweight per-IP rate limit inside the route handler (reuse
    the existing ratelimit module pattern from /api/holds or /api/bookings/initiate).
    Suggested: 20 req/min per IP (covers 12 req/min polling at 5s intervals with
    headroom).

================================================================================
BACK-OF-ENVELOPE: Polling load at target scale
================================================================================

  Assumptions: 100 concurrent bank transfer payments at peak.
  Current: 2 DB queries per 5s poll = 24 req/min/client = 2,400 req/min total.
  After P1 fix: 1 DB query per 5s poll = 12 req/min/client = 1,200 req/min total.

  Each query is a single-row index lookup (unique B-tree). PostgreSQL handles
  50,000+ simple index lookups/sec on modest hardware. 1,200/min = 20/sec.
  Even at 2,400/min = 40/sec (unfixed), this is well within capacity.

  Conclusion: functionally safe at target scale. The P1 fix halves unnecessary
  load and should be done for hygiene, but this is not a scaling cliff.
