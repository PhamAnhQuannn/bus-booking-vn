# Technical Debt Scan
Date: 2026-06-12
Scope: Full codebase refresh (prior scan: 2026-06-05; PRs 12, 15, 16 landed since)

## Summary

Overall debt level is **low**. The codebase is tightly maintained with enforced linting boundaries, full TypeScript strict mode, and consistent use of `withErrorHandler` across the 109 customer- and operator-facing routes (100% coverage). The highest-risk gap is the three `op/reports/*` route files that skip `withErrorHandler` but chain `requireOperatorAuth` directly — any unhandled DB throw in those handlers leaks as an unformatted 500 with no Pino log entry. Secondary concerns are the four storage and two Sentry TODO stubs that are pre-conditions for go-live (issue 094), and one intentional `TEMP` bypass in admin TOTP that needs cleanup before release.

## P2 — Medium Debt (maintenance burden)

**1. `op/reports/*` routes missing `withErrorHandler` — unlogged 500s on DB throws**

Three report routes skip `withErrorHandler` and chain `requireOperatorAuth` directly with no outer error catch:

- `app/api/op/reports/revenue/route.ts` — line 63: `export const GET = requireOperatorAuth({})(getHandler);`
- `app/api/op/reports/revenue.csv/route.ts` — line 76: `export const GET = requireOperatorAuth({})(getHandler);`
- `app/api/op/reports/payouts/route.ts` — line 22: `export const GET = requireOperatorAuth({})(getHandler);`

Risk: if `getRevenueReport`, `getBookingRevenueRows`, or `getPayoutReport` throw (e.g. Prisma connectivity failure), the error surfaces as an unformatted Next.js 500 with no Pino log entry and no `{ error: 'Internal server error' }` shape. This breaks the uniform error contract the client expects.

Fix effort: 15 min — wrap each export in `withErrorHandler(requireOperatorAuth({})(handler))` (one line change per file, mirrors the finance `financeRoute()` pattern).

**2. `op/reports/payouts/[id]/retry` uses `withErrorHandler` inline rather than as outer wrapper**

`app/api/op/reports/payouts/[id]/retry/route.ts` lines 28–56: the exported `POST` is NOT wrapped by `withErrorHandler`; instead `withErrorHandler` is called inside the function body and wraps only the inner closure. The result is correct but non-idiomatic — a reviewer reading only the export line `export async function POST(...)` sees no error guard.

Fix effort: 10 min — refactor to the standard HOF pattern: `export const POST = withErrorHandler(requireOperatorAuth({})(handler));`

**3. Admin finance and moderation routes: `withErrorHandler` applied via `_shared.ts` HOF — invisible at the export line**

`app/api/admin/finance/*` (6 routes) use `financeRoute(handler)` which internally calls `withErrorHandler`. `app/api/admin/moderation/*` (5 routes) use `moderationRoute(handler)` similarly. This is architecturally correct — the `_shared.ts` comment explains it — but the pattern means `withErrorHandler` is NOT visible in the route file itself. New contributors adding a finance or moderation route could easily skip the HOF and not realise they also lost the error guard.

Fix effort: 0 code change needed; add a one-line comment at the top of each `_shared.ts` explicitly documenting that `withErrorHandler` is baked in, so the convention is discoverable. Already partially documented in `app/api/admin/finance/_shared.ts:9`.

**4. Cron routes: manual `try/catch` instead of `withErrorHandler` — inconsistent stack-trace logging**

All 11 cron routes (`app/api/cron/*`) implement their own inline `try/catch` with `logger.error({ err }, ...)`. This is functional but the log format differs slightly across files (some log `result`, others `err`, some include the cron name in the message). A unified `withErrorHandler` or a `cronRoute(handler)` HOF (analogous to `financeRoute`) would standardise the error shape.

Fix effort: medium — create a `cronRoute(cronName: string, handler: ...)` HOF in a `_shared.ts` parallel to the admin HOFs. Not urgent before launch.

## Code Duplication Hot Spots

**1. Inline `requireOperatorAuth({})` at route export — 70+ occurrences**

Every `app/api/op/**` route wraps its export with `requireOperatorAuth({})(handler)` directly. The admin routes solved this with per-area `_shared.ts` HOFs (`financeRoute`, `moderationRoute`). An `opRoute(handler)` HOF in `app/api/op/_shared.ts` would centralise operator auth composition and make adding a cross-cutting concern (e.g. rate limiting) a one-file change instead of 70.

**2. Date-pattern Zod regex duplicated across 4 report routes**

`/^\d{4}-\d{2}-\d{2}$/` appears in at least:
- `app/api/op/reports/revenue/route.ts:24`
- `app/api/op/reports/revenue.csv/route.ts`
- `app/api/op/bookings/route.ts`
- `app/api/op/trips/route.ts`

Should be extracted to `lib/core/validation/schemas.ts` (or equivalent) as `dateStringSchema`.

**3. `payoutIdFromUrl` / `idFromUrl` segment-parsing helpers duplicated in `_shared.ts` files**

The pattern of splitting `req.nextUrl.pathname` and locating a segment appears in both `app/api/admin/finance/_shared.ts` and `app/api/admin/moderation/_shared.ts`. Low duplication count (2 files) but the pattern could be a shared `segmentFromUrl(req, after: string)` utility in `lib/core/routing/`.

**4. Cron auth header check duplicated across 11 files**

Each cron route independently implements:
```ts
const authHeader = req.headers.get('authorization');
const cronSecret = process.env.CRON_SECRET;
if (cronSecret && authHeader !== `Bearer ${cronSecret}`) {
  return NextResponse.json({ error: 'UNAUTHORIZED' }, { status: 401 });
}
```
This is an ideal candidate for the proposed `cronRoute()` HOF (P2-4 above).

**5. `SELECT FOR UPDATE` pattern in service functions**

The advisory-lock SELECT pattern (`tx.$queryRaw\`SELECT id FROM "..." WHERE ... FOR UPDATE\``) appears in at least: `lib/trips/reassignBus.ts`, `lib/trips/cancelTrip.ts`, `lib/trips/createTrip.ts`, `lib/buses/updateBus.ts`, `lib/ledger/retryPayout.ts`. Each hand-codes the raw SQL. A typed helper `lockRow(tx, table, id)` would reduce surface area for the `::uuid` cast mistake (documented in Mistake Log 2026-05-19).

## Test Debt

**Skipped E2E tests (32 instances across 17 files):** All skips are conditional `test.skip(condition, reason)` guards — none are unconditional `test.skip` modifiers. Two categories:

1. **Sandbox-gate skips (17):** Controlled by `E2E_*_ENABLED` env vars. These are intentional — the sandbox gates prevent specs from running against a live DB without explicit opt-in. Not debt; by design.

2. **Data-availability skips (9 in hold-flow + momo-booking):** Tests that skip when prior test steps produced no data (e.g. `test.skip(!holdId, ...)`). These are defensive guards for cascading-failure isolation, not test debt. However, the `momo-booking.spec.ts` file has 7 data-gated skips — if the hold creation step fails, almost all downstream tests are silent. Consider converting to `test.fail` expectations for clearer signal.

3. **Deferred tests (2):** `hold-flow.spec.ts` WebKit mobile flake (issues/102) and MoMo sandbox Phase 5 — both have issue references and are not forgotten.

**`setTimeout` in tests (3 instances):** All three are intentional lock-wait / window-expiry patterns in integration tests (`cronJobs.int.test.ts:554,660` — 250ms lock wait; `factory.test.ts:63` — 100ms rate-limit window). Not timing-flake debt.

**No `.skip`, `xit`, `xdescribe`, or `xtest` modifiers found** — the test suite has no silently-skipped tests.

## Recommendations

1. **[15 min, P2]** Fix the three `op/reports/*` GET routes — wrap with `withErrorHandler`. Copy the `revenue/route.ts` pattern but add the outer wrapper. Prevents silent unlogged 500s on DB errors for operators running reports.

2. **[10 min, P2]** Refactor `op/reports/payouts/[id]/retry/route.ts` from inline `withErrorHandler` to the standard HOF pattern. Makes the route's error contract visible at the export line.

3. **[30 min, P2]** Create a `cronRoute(jobName, handler)` HOF in `app/api/cron/_shared.ts` to DRY the 11 repeated auth-header + try/catch patterns. Returns consistent `{ rowsAffected, status }` shape on success, `{ error: 'UNAUTHORIZED' }` on 401, `{ error: 'internal_error' }` on throw. Block on 094 timeline — this is a nice-to-have pre-launch cleanup.

4. **[P3, before 094]** Remove `lib/auth/totpDisabled.ts` and its 2 call sites. The TOTP bypass has no place in a go-live build even with the production guard, as it creates a false sense of security in staging-but-production-NODE_ENV environments.

5. **[P3, extract]** Move the `/^\d{4}-\d{2}-\d{2}$/` regex into `lib/core/validation/schemas.ts` as `dateStringSchema`. Small change; eliminates 4 copy-pastes and prevents future drift (e.g. one copy accepting `YYYY/MM/DD` by mistake).

6. **[P3, wave9]** The 4 S3 TODO stubs, 2 Sentry stubs, and 1 refund stub are go-live pre-conditions already tracked by milestone tags. No action needed beyond ensuring 094 scope includes them.
