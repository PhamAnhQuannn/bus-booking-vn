## P1 — WCAG AA Violations (must fix)

### P1-01 · Admin login: inputs unlabelled
**File:** `app/admin/login/page.tsx` lines 86–94, 95–103, 110–119
**WCAG:** 1.3.1 Info and Relationships; 1.1.1 Non-text Content; 4.1.2 Name, Role, Value
**Issue:** All three inputs (email, password, TOTP code) use `placeholder` as their only label. Placeholders disappear on typing, fail users with cognitive disabilities, and are not reliably announced by screen readers as persistent labels.
**Fix:** Add a visually-hidden `<label>` paired via `htmlFor` to each input, e.g.:
```tsx
<label htmlFor="admin-email" className="sr-only">Email</label>
<input id="admin-email" type="email" … />
```
Or replace bare `<input>` with the design-system `<Label>` + `<Input>` pair already used in every other form in this codebase.

---

### P1-02 · Admin login: error message not announced to screen readers
**File:** `app/admin/login/page.tsx` line 82
**WCAG:** 4.1.3 Status Messages
**Issue:** `{error && <p className="mt-3 text-sm text-red-600">{error}</p>}` — the `<p>` is inserted into the DOM dynamically after a failed login, but has no `role="alert"` or `aria-live` attribute. Screen readers will not announce it.
**Fix:** Add `role="alert"` (assertive announcement) to the error paragraph, matching the pattern used in every other form in the codebase:
```tsx
{error && <p role="alert" aria-live="assertive" className="mt-3 text-sm text-red-600">{error}</p>}
```

### P1-03 · Orange primary on white fails WCAG AA 4.5:1 for normal text
**File:** `app/globals.css` line 79
**WCAG:** 1.4.3 Contrast (Minimum)
**Issue:** `--primary: oklch(0.646 0.222 41.116)` — the CSS comment on line 78 itself states "White label on orange-600 ≈ 3:1 (fine for the **bold** Button label)". This is only acceptable for large text (≥18pt/24px, or bold ≥14pt/18.67px) per WCAG 1.4.3. Normal-size uses of `text-primary` (e.g. navigation links in `SiteHeader`, active nav links at 14px, price display in search results at `text-xl` mono) fall below 4.5:1. The link colour `text-primary` on links inside paragraphs (register/login pages, footer) at small size definitively fails.
**Specific offending sites:**
- `app/(customer)/auth/login/page.tsx` lines 93–101 — "Quên mật khẩu?" and "Đăng ký" links at `text-sm` (14px)
- `app/(customer)/auth/register/page.tsx` line 309 — "Đăng nhập" link at `text-sm`
- `components/layout/SiteHeader.tsx` line 45 — active nav item `text-primary` at `text-sm`
- `components/layout/SiteFooter.tsx` line 52 — all nav links at `text-sm`
- `app/(customer)/search/page.tsx` line 141 — price display `text-primary` (large, bold — likely passes for large-text threshold, but depends on rendered size)
**Fix:** Deepen the primary hue toward orange-700 (`oklch(0.55 0.20 41)` ≈ 4.6:1 on white) for all text applications. Button labels at large/bold weight can retain orange-600. The fix should define a `--primary-text` token for text-only uses while keeping `--primary` for backgrounds. A targeted short-term fix is to replace `text-primary` on links with a slightly darker token.

### P1-04 · `NewTripClient`: Route and Bus `<Label>` elements have no `htmlFor` — not linked to their Select triggers
**File:** `app/op/(console)/trips/new/NewTripClient.tsx` lines 201, 222
**WCAG:** 1.3.1 Info and Relationships; 4.1.2 Name, Role, Value
**Issue:** The "Tuyến đường" and "Xe" labels are `<Label>` components with no `htmlFor` prop, and the `<Select>` / `<SelectTrigger>` has no matching `id`. The programmatic association between label and control is absent — screen readers cannot announce the label when the trigger is focused.
```tsx
<Label>Tuyến đường</Label>       {/* no htmlFor */}
<Select …><SelectTrigger …>      {/* no id */}
```
The same pattern occurs for "Loại xe" in `BusesClient.tsx` (line 275) and "Khu vực đón khách" (line 283).
**Fix:** Add `id` to `SelectTrigger` and `htmlFor` to `Label`. Since base-ui's `SelectTrigger` renders a `<button>`, the `htmlFor` trick won't work — instead use `aria-label` or `aria-labelledby` on the trigger:
```tsx
<Label id="route-label">Tuyến đường</Label>
<SelectTrigger aria-labelledby="route-label" …>
```

### P1-06 · `HoldExpiryModal`: Non-dismissible modal may trap keyboard focus without escape
**File:** `components/HoldExpiryModal.tsx` lines 39–52
**WCAG:** 2.1.2 No Keyboard Trap
**Issue:** The modal is `open={isExpired}` with no `onOpenChange` handler (intentionally non-dismissible) and `showCloseButton={false}`. The comment says "Esc/backdrop cannot close it." While base-ui's Dialog primitive normally manages Esc-to-close, removing the handler and the close button means keyboard users who accidentally end up on this dialog cannot escape it by standard means. The auto-redirect after expiry is not a keyboard-accessible mitigation — there is no announced countdown or clear instruction for keyboard users. The Dialog uses base-ui which manages focus trap correctly; however the lack of any escape route is a concern.
**Mitigating factor:** The button "Tìm chuyến xe mới" is keyboard-reachable within the dialog, so focus is not permanently trapped. However the Esc key being disabled is still a WCAG 2.1.2 issue.
**Fix:** Either re-enable Esc (which routes to search — same as the button), or ensure the dialog description includes the redirect timing so AT users know what will happen: `aria-describedby` linking to text like "Sẽ tự chuyển sang tìm kiếm sau 5 giây."

## P2 — Should Fix

### P2-01 · Admin login: submit button has no visible focus ring
**File:** `app/admin/login/page.tsx` line 104
**WCAG:** 2.4.11 Focus Appearance (WCAG 2.2)
**Issue:** The submit button uses a raw `className="w-full rounded bg-primary px-3 py-2 text-white"` without any `focus-visible:ring-*` class. All other buttons in the codebase use the design-system `<Button>` component which has `focus-visible:ring-3 focus-visible:ring-ring/50`. This bare `<button>` class string provides only browser-default focus styles (which may be insufficient depending on browser/OS).
**Fix:** Replace with the design-system `<Button>` component and `<Label>`+`<Input>` (which also fixes P1-01 and P1-02):
```tsx
import { Button } from '@/components/ui/button';
<Button type="submit" disabled={busy} className="w-full">Tiếp tục</Button>
```

### P2-02 · `NewTripClient`: Pickup area group header is a `<div>`, not semantic
**File:** `app/op/(console)/trips/new/NewTripClient.tsx` line 337
**WCAG:** 1.3.1 Info and Relationships
**Issue:** The group heading "BẾN XE" / "ĐÓN TẬN NƠI" uses a plain `<div>` styled as a header rather than a semantic `<legend>` or accessible heading. The checkbox group has no `<fieldset>` wrapper — the group label is purely presentational.

```tsx
<fieldset>

</fieldset>
```

### P2-03 · `SearchPage` disabled date chip uses `<span>` with only `aria-disabled`
**File:** `app/(customer)/search/page.tsx` lines 193–200 and 291–298
**WCAG:** 4.1.2 Name, Role, Value
**Issue:** The "← Trước" chip when previous date is disabled renders as a `<span aria-disabled="true">`. A `<span>` has no implicit interactive role, so `aria-disabled` on a non-interactive element is meaningless — screen readers will not communicate "disabled" to users because they have no reason to visit a plain `<span>`. The element also has no `tabIndex` or `role`, so it is not keyboard-reachable at all. While making it unreachable is one valid approach, the sighted visual treatment (same visual weight as the active chip) suggests users may expect to interact with it.
**Fix:** Either omit the disabled chip entirely (simplest), or render it as a `<button disabled aria-label="…">` which correctly communicates disabled state to AT.

### P2-04 · `OperatorNav` mobile drawer: `Dialog.Title` wraps `<Logo>` — visual title absent for AT
**File:** `components/op/OperatorNav.tsx` lines 280–284

**Issue:** The mobile drawer's `<Dialog.Title>` contains only a logo graphic (`<Logo variant="combo" />`). If `Logo` renders as a `<svg>` or `<img>` without descriptive alt text, the dialog title announced to screen readers may be empty or meaningless.

### P2-09 · `PopularTrips` carousel: `role="region"` on `<ul>` conflicts with list semantics
**File:** `components/home/PopularTrips.tsx` line 64
**WCAG:** 1.3.1 Info and Relationships
**Issue:** The carousel `<ul>` has `role="region"` and `aria-label="Tuyến phổ biến"`. Adding `role="region"` overrides the implicit `list` role — the element is no longer a list to AT. This means screen reader virtual cursor may not see the `<li>` children as list items.
**Fix:** Either keep `role="list"` (default for `<ul>`) and wrap in a separate `<section aria-label="Tuyến phổ biến">`, or use `role="group"` on the `<ul>` which preserves the list browsing mode in some AT. The section wrapper is cleaner:
```tsx
<section aria-label="Tuyến phổ biến">
  <ul className="…">…</ul>
</section>
```

### P2-10 · `Calendar` weekday abbreviations: inaccessible to screen readers
**File:** `components/ui/calendar.tsx` lines 147–156
**WCAG:** 1.3.1 Info and Relationships
**Issue:** Weekday headers "T2"–"CN" are plain `<div>` elements with no `scope` or semantic meaning. Screen readers reading the grid will encounter these abbreviations without context.
**Fix:** Use `<th scope="col">` for weekday headers (requires `<table>`/`<thead>`/`<tr>` structure), or add `role="columnheader"` to each header `<div>`. The full weekday name can be in a `<span className="sr-only">` next to the abbreviation:
```tsx
<div role="columnheader" aria-label="Thứ Hai">T2</div>
```

## P3 — Best Practice

### P3-01 · `CustomerForm` field errors: no `aria-invalid` on inputs
**File:** `app/(customer)/booking/customer/CustomerForm.tsx` lines 252–265, 271–285, 288–300
**Issue:** When field errors exist, the error paragraph is linked via `aria-describedby`, which is correct. However the input itself does not receive `aria-invalid="true"`. The design-system `<Input>` component supports `aria-invalid` (it has `aria-invalid:border-destructive` styling). Without `aria-invalid`, AT users must read the description to discover the error — `aria-invalid` is the primary signal that a field is in error state.
**Fix:** Add `aria-invalid={!!fieldErrors.buyerName || undefined}` to each input.

### P3-02 · `ReviewClient` consent checkboxes: no `aria-required` or error announcement
**File:** `app/(customer)/booking/review/ReviewClient.tsx` lines 201–225

## Portal Summary

### Customer Portal
**Overall quality: Good.** The customer-facing auth forms (login, register, forgot-password) use the design-system `<Label>`+`<Input>` pair with correct `htmlFor`/`id` pairing. Error messages consistently use `role="alert" aria-live="assertive"`. Buttons use the design-system `<Button>` with `focus-visible:ring-*`. The booking customer form (`CustomerForm.tsx`) correctly uses `<fieldset>/<legend>` for the pickup section, `aria-describedby` for error messages, and `role="alert"` for submission errors. The `HoldTimer` is partially accessible (good `aria-label` on wrapper, but emoji not hidden — P2-07). The search page has solid structure with `<main>`, `<article>` for trip cards, and `aria-live="polite"` on result count. **Primary gaps:** orange-on-white contrast (P1-03), `aria-invalid` missing on field inputs (P3-01), disabled date chip as `<span>` (P2-03), emoji not hidden in HoldTimer (P2-07), no skip link in root layout (P2-06).

### Operator Portal
**Overall quality: Good.** Auth pages (login, register, first-login, forgot-password) use the design-system components with proper labelling. The `NewTripClient` form has labelled inputs for departure/price but unlabelled Select triggers for route/bus (P1-04). `BusesClient` has unlabelled capacity inputs in table cells (P1-05) and the "Loại xe" select without `htmlFor` linkage (P1-04). The `ConfirmDialog` uses base-ui Dialog with proper `DialogTitle`, `DialogDescription`, focus trap, and Esc-close. `OperatorNav` has proper skip link, `aria-label` on navs, `aria-current="page"`, and correct focus ring styles. `DataTable` has `<caption className="sr-only">`, `scope="col"` on headers, `aria-busy` during loading, and an `aria-live="polite"` status region. **Primary gaps:** unlabelled Select controls (P1-04), unlabelled capacity inputs in table (P1-05), `aria-expanded` without `aria-controls` on expand buttons (P2-08), mobile drawer Dialog.Title containing only logo (P2-04).

### Admin Portal
**Overall quality: Poor.** The admin login page (`app/admin/login/page.tsx`) has all three design-review P1 violations confirmed: no labels on inputs (P1-01), no `role="alert"` on error (P1-02), and the submit button bypasses the design system with no focus ring (P2-01). The admin console layout is missing the skip link present in the operator console (P2-05). The `AdminNav` itself is well-structured: `aria-label="Admin console"`, `aria-current="page"` on active links, correct focus rings. The admin console `<aside>` has a title `<span>` but no semantic heading — consider `<h2 className="…">` for the sidebar title "Bảng điều khiển".
