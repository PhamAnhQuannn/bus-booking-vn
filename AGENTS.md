## Mistake Log

- **2026-05-17 (Issue 001)** — Maintenance-window filter compared `new Date()` (wall-clock now) instead of `trip.departureAt`. Future trips on a bus with a future maintenance window were incorrectly included whenever maintenance wasn't active *right now*. Fix: window-vs-window overlap (`maintenanceStart ≤ endUtc AND maintenanceEnd ≥ startUtc`) — encoded as `OR: [maintenanceStart null, maintenanceEnd < startUtc, maintenanceStart > endUtc]` in `app/api/trips/search/route.ts` + `lib/db/searchTrips.ts`.
- **2026-05-17 (Issue 001)** — `searchResultSelect` whitelisted `salesClosed` + `status` despite both being `where`-clause-only. Leaked internal state into client payload. Fix: drop from select; filter logic in `where` unchanged. Rule: `select` whitelist = exactly the UI contract fields, no filter columns.
- **2026-05-17 (Issue 001)** — Seed phone placeholders `+84901111111` matched `.gitleaks.toml` regex `\+84[35789]\d{8}` (`9` is a valid mobile prefix). Real-mobile-prefix all-digits placeholders trip the very rule meant to catch real numbers. Fix: use literal-x mask `+8490xxxxxx[12]` — `\d{8}` can't consume `x` chars. Rule: PII placeholders must escape the project's PII detection regex.
- **2026-05-17 (Issue 001)** — Phase 4b plan referenced a `getRateLimiter().check()` API that didn't exist; real API is `ratelimit.limit(ip)` returning `{allowed, remaining, retryAfter}`. Always read the actual module before writing mocks in the plan.
- **2026-05-17 (Issue 002)** — Server component `app/booking/review/page.tsx` did a server-side fetch to its own API using `process.env.NEXT_PUBLIC_BASE_URL ?? 'http://localhost:3000'`. Broke whenever `next dev` auto-bumped to 3001 (port 3000 occupied by another app) — fetch hit the wrong origin, response wasn't JSON, page threw 500. Fix: derive base URL from incoming request headers — `const h = await headers(); const baseUrl = ` `${h.get('x-forwarded-proto') ?? 'http'}://${h.get('host') ?? 'localhost:3000'}` ``. Rule: server components calling own routes must derive origin from the request, never from a static env fallback. Better still — call the underlying lib function in-process rather than self-fetching.
- **2026-05-17 (Issue 002)** — Playwright e2e driving `@base-ui/react` `<Input>` via `fill()` / `pressSequentially()` was flaky across cold workers: the primitive owns `onChange` internally and the public change API is `onValueChange`, so keystrokes route through a spread `onChange` prop with timing-sensitive merge behavior. Fix: bypass the form in specs that aren't testing the form — `page.goto('/search?' + URLSearchParams)` lands directly on the RSC results page. Rule: drive form-incidental e2e via URL state, not synthetic keystrokes on third-party-controlled inputs.
- **2026-05-17 (Issue 003)** — Issue 003 carried `app/booking/review/page.tsx` forward with a server-side `fetch('${baseUrl}/api/holds/[id]')` even though the Issue 002 Mistake Log already said "Better still — call the underlying lib function in-process rather than self-fetching." Static-review caught the residual self-fetch. Fix: extracted `lib/booking/getHoldDetails.ts` (pure DB read); both `app/api/holds/[id]/route.ts` and `app/booking/review/page.tsx` now verify the bb_hold cookie then call the lib in-process. Rule (hardened): the "Better still" clause of the Issue 002 entry is now mandatory — server components MUST NOT self-fetch their own API. Extract a lib function and call it from both the route handler and the server component. Header-derived base URLs are not an acceptable long-lived solution.
- **2026-05-17 (Issue 003)** — `e2e/cash-booking.spec.ts` asserted booking-ref shape with `/BB-\d{4}-[A-Z0-9]{4}-[A-Z0-9]{4}/` (uppercase). `lib/booking/bookingRef.ts` emits lowercase base36 (`[0-9a-z]{4}`); the assertion would have failed against any real booking row. Static review caught it — no live Playwright run had executed. Fix: e2e regex now `/BB-\d{4}-[0-9a-z]{4}-[0-9a-z]{4}/` and matches the exported `BOOKING_REF_REGEX`. Rule: assertions on generator output should reference the generator's exported regex (`BOOKING_REF_REGEX`) rather than re-typing the pattern — re-typed patterns drift in case and width.
- **2026-05-18 (Issue 004)** — MoMo webhook `FAILURE_RESULT_CODES` set was widened beyond the Issue 004 acceptance spec — the implementer added `1006/1007/1026/9001` from general MoMo docs knowledge. `9001` in particular collides with the "pending" pattern (`9000` is pending; `9001` is "transaction in process" in some MoMo docs revisions) and would have mis-transitioned in-flight payments to `payment_failed_expired`. Static QA caught the superset before any live IPN routed through it. Fix: pinned `FAILURE_RESULT_CODES` to the spec's exact list `{1001, 1002, 1003, 1004, 1005, 4100}` in `app/api/payments/momo/webhook/route.ts`. Rule: enum/code-list constants that drive state-machine transitions must be sourced from the issue acceptance criteria verbatim, not augmented from upstream vendor docs. Vendor-doc supersets belong in a follow-up issue with explicit AC additions, never silently merged into the original implementation.
