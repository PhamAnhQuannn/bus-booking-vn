## Mistake Log

- **2026-05-17 (Issue 001)** — Maintenance-window filter compared `new Date()` (wall-clock now) instead of `trip.departureAt`. Future trips on a bus with a future maintenance window were incorrectly included whenever maintenance wasn't active *right now*. Fix: window-vs-window overlap (`maintenanceStart ≤ endUtc AND maintenanceEnd ≥ startUtc`) — encoded as `OR: [maintenanceStart null, maintenanceEnd < startUtc, maintenanceStart > endUtc]` in `app/api/trips/search/route.ts` + `lib/db/searchTrips.ts`.
- **2026-05-17 (Issue 001)** — `searchResultSelect` whitelisted `salesClosed` + `status` despite both being `where`-clause-only. Leaked internal state into client payload. Fix: drop from select; filter logic in `where` unchanged. Rule: `select` whitelist = exactly the UI contract fields, no filter columns.
- **2026-05-17 (Issue 001)** — Seed phone placeholders `+84901111111` matched `.gitleaks.toml` regex `\+84[35789]\d{8}` (`9` is a valid mobile prefix). Real-mobile-prefix all-digits placeholders trip the very rule meant to catch real numbers. Fix: use literal-x mask `+8490xxxxxx[12]` — `\d{8}` can't consume `x` chars. Rule: PII placeholders must escape the project's PII detection regex.
- **2026-05-17 (Issue 001)** — Phase 4b plan referenced a `getRateLimiter().check()` API that didn't exist; real API is `ratelimit.limit(ip)` returning `{allowed, remaining, retryAfter}`. Always read the actual module before writing mocks in the plan.
- **2026-05-17 (Issue 002)** — Server component `app/booking/review/page.tsx` did a server-side fetch to its own API using `process.env.NEXT_PUBLIC_BASE_URL ?? 'http://localhost:3000'`. Broke whenever `next dev` auto-bumped to 3001 (port 3000 occupied by another app) — fetch hit the wrong origin, response wasn't JSON, page threw 500. Fix: derive base URL from incoming request headers — `const h = await headers(); const baseUrl = ` `${h.get('x-forwarded-proto') ?? 'http'}://${h.get('host') ?? 'localhost:3000'}` ``. Rule: server components calling own routes must derive origin from the request, never from a static env fallback. Better still — call the underlying lib function in-process rather than self-fetching.
- **2026-05-17 (Issue 002)** — Playwright e2e driving `@base-ui/react` `<Input>` via `fill()` / `pressSequentially()` was flaky across cold workers: the primitive owns `onChange` internally and the public change API is `onValueChange`, so keystrokes route through a spread `onChange` prop with timing-sensitive merge behavior. Fix: bypass the form in specs that aren't testing the form — `page.goto('/search?' + URLSearchParams)` lands directly on the RSC results page. Rule: drive form-incidental e2e via URL state, not synthetic keystrokes on third-party-controlled inputs.
- **2026-05-17 (Issue 003)** — Issue 003 carried `app/booking/review/page.tsx` forward with a server-side `fetch('${baseUrl}/api/holds/[id]')` even though the Issue 002 Mistake Log already said "Better still — call the underlying lib function in-process rather than self-fetching." Static-review caught the residual self-fetch. Fix: extracted `lib/booking/getHoldDetails.ts` (pure DB read); both `app/api/holds/[id]/route.ts` and `app/booking/review/page.tsx` now verify the bb_hold cookie then call the lib in-process. Rule (hardened): the "Better still" clause of the Issue 002 entry is now mandatory — server components MUST NOT self-fetch their own API. Extract a lib function and call it from both the route handler and the server component. Header-derived base URLs are not an acceptable long-lived solution.
- **2026-05-17 (Issue 003)** — `e2e/cash-booking.spec.ts` asserted booking-ref shape with `/BB-\d{4}-[A-Z0-9]{4}-[A-Z0-9]{4}/` (uppercase). `lib/booking/bookingRef.ts` emits lowercase base36 (`[0-9a-z]{4}`); the assertion would have failed against any real booking row. Static review caught it — no live Playwright run had executed. Fix: e2e regex now `/BB-\d{4}-[0-9a-z]{4}-[0-9a-z]{4}/` and matches the exported `BOOKING_REF_REGEX`. Rule: assertions on generator output should reference the generator's exported regex (`BOOKING_REF_REGEX`) rather than re-typing the pattern — re-typed patterns drift in case and width.
- **2026-05-18 (Issue 004)** — MoMo webhook `FAILURE_RESULT_CODES` set was widened beyond the Issue 004 acceptance spec — the implementer added `1006/1007/1026/9001` from general MoMo docs knowledge. `9001` in particular collides with the "pending" pattern (`9000` is pending; `9001` is "transaction in process" in some MoMo docs revisions) and would have mis-transitioned in-flight payments to `payment_failed_expired`. Static QA caught the superset before any live IPN routed through it. Fix: pinned `FAILURE_RESULT_CODES` to the spec's exact list `{1001, 1002, 1003, 1004, 1005, 4100}` in `app/api/payments/momo/webhook/route.ts`. Rule: enum/code-list constants that drive state-machine transitions must be sourced from the issue acceptance criteria verbatim, not augmented from upstream vendor docs. Vendor-doc supersets belong in a follow-up issue with explicit AC additions, never silently merged into the original implementation.
- **2026-05-18 (Issue 007)** — Issue 007 migration hand-patched 5 indices via raw SQL (`Customer_email_key` partial, `OtpAttempt_phone_active_key` partial unique, `OtpAttempt_phone_createdAt_idx` composite DESC, `Session_customerId_idx`, `Booking_customerId_idx`) but did NOT declare the three non-partial ones in `prisma/schema.prisma`. On the next `prisma migrate dev` invocation Prisma detected schema-vs-DB drift and prompted interactively for a follow-up migration name — breaking unattended autopilot dispatch. Fix: declared the 3 non-partial indices as `@@index` in `schema.prisma` so the raw SQL and the schema agree; the 2 partial indices (`WHERE` clauses) stay SQL-only because Prisma's DSL can't model them. Rule: any raw-SQL index that *could* be expressed in the Prisma DSL (plain composite, plain single-column on a non-`@unique` column) MUST also be declared in `schema.prisma` in the same migration. Partial / expression / `WHERE`-clause indices are the only exception — those stay SQL-only and Prisma silently ignores them in diff. Verify with `pnpm prisma migrate diff --from-schema-datamodel prisma/schema.prisma --to-schema-datasource prisma/schema.prisma` after every hand-patched migration.
- **2026-05-18 (Issue 007)** — Phase 3 plan handed the verified-phone state from `/api/auth/otp/verify` to `/api/auth/register` via a server-side session table lookup keyed on phone. That coupled two distinct flows to one DB row and required a second round-trip just to read state the verify route had in-hand. Iter-2 lead pivoted to a single-use `otpProof` JWT (HS256, 5min TTL, `purpose: 'register'|'login'`, claims `{phone, jti}`) returned in the verify response body and re-presented in the register/login body. Register/login `jwt.verify` + jti-consume (Redis SETNX) makes the proof one-shot; logger redaction list extended to include `otpProof`. Rule: when verified state must cross a route boundary, prefer a short-TTL signed proof JWT over a server-session row — fewer DB writes, no cross-route table coupling, replay-safe via jti. Add the proof field to the top-level logger redact list the same commit it's introduced (else it leaks through structured logs).
- **2026-05-18 (Issue 007)** — Adding the CSRF double-submit middleware in `proxy.ts` for non-safe `/api/*` blocked every existing state-changing client + e2e caller that wasn't auth-related. Audit at Phase 5 found `lib/api/holdsClient.ts` POST `/api/holds`, `app/booking/review/ReviewClient.tsx` POST `/api/bookings/initiate`, and 4 Playwright POSTs across `hold-flow.spec.ts` + `momo-booking.spec.ts` would all 403 once the middleware shipped — none of these touch auth. Fix: extracted `lib/auth/csrfClient.ts` (browser `readCsrfToken()` reading non-HttpOnly `bb_csrf` cookie) and `e2e/helpers/csrf.ts` `primeCsrf()` (Playwright cookie extraction via `request.storageState()`); threaded `X-CSRF-Token` header into all 6 call sites. Webhook routes that authenticate via HMAC (`/api/payments/momo/webhook`) stay exempted in the middleware. Rule: when adding a cross-cutting request-gate middleware, grep every non-safe-method caller of `/api/*` (both `fetch(` in `app/**` + `lib/**` and `request.post(` / `request.put(` in `e2e/**`) BEFORE merging, not after. Build the client-side helper in the same commit as the middleware — a "fix the callers later" plan strands the codebase in a broken state between merges.
- **2026-05-18 (Issue 010)** — Iter-1 operator-session tests used non-hex string literals (`'wronghash'`, `'correcthash'`) as mock SHA-256 hash inputs to `Buffer.from(str, 'hex')` + `crypto.timingSafeEqual`. Non-hex chars decode to empty 0-byte buffers — `timingSafeEqual(Buffer.alloc(0), Buffer.alloc(0))` returns `true`, so every "wrong hash" assertion silently passed and the test gave false confidence in reuse-detection. Fix: replace with valid 64-char hex literals (`'a'.repeat(64)` and `'b'.repeat(64)`); `timingSafeEqual` then correctly compares 32-byte buffers and the mismatch path fires as expected. Rule: any test that passes a string literal to `Buffer.from(s, 'hex')` MUST use a hex-valid string of the expected byte-width (64 chars for SHA-256). Codify with a `hexMock(byte: 'a' | 'b', len = 32) => byte.repeat(len * 2)` helper if reused across files.
- **2026-05-18 (Issue 010)** — AC4 ("3 failed OTP verifications → 15-min lockout") was initially implemented as a count-based check tied to the 5-min OTP TTL — the row expired before the 15-min window could mean anything. Fix: reuse the existing OTP row as a **lockout sentinel** — on the 3rd verify mismatch, set `consumed=true` AND `expiresAt = now + 15min` (extending the row's life past the OTP TTL for sentinel purposes only). `findLockoutSentinel(phone)` queries `consumed=true AND attemptCount>=3 AND expiresAt>NOW()` and is called BEFORE both the send-OTP path and the verify-OTP path. No new column, no new table. The same row carries two semantics: active OTP (when `consumed=false AND expiresAt>NOW`) vs lockout marker (when `consumed=true AND attemptCount>=3 AND expiresAt>NOW`). Rule: when a lockout window outlasts the natural lifetime of the gate-row, extend the row's `expiresAt` past its original TTL and flip a state column to repurpose it as a sentinel — avoids a parallel lockout table for a single boolean state. Document the dual-meaning in the model comment so future readers don't "clean up" the orphan rows.
- **2026-05-18 (Issue 010)** — Initial implementation enforced `requiresPasswordChange` only at the per-route layer (each `/api/op/*` handler called `requireOperatorAuth` which checked the flag). QA flagged: no layer-level guard meant a new `/op/*` page added later could silently skip the gate. The DB-read-per-request approach also broke Edge runtime — `proxy.ts` (middleware) can't query Prisma. Fix: add `requiresPasswordChange` claim INSIDE the operator access JWT itself; middleware verifies the token via `jose.jwtVerify` (Edge-safe) and reads the claim with zero DB hits; redirect to `/op/first-login` when true. Allowlist is exact-match `Set` of paths (NOT `startsWith` — prevents `/op/first-login-bypass` style sneak-throughs). The flag-clear path on password-change route MUST mint a fresh token with `requiresPasswordChange: false` in the same transaction as the DB update, else the old JWT keeps redirecting. Rule: cross-cutting gates that must run in Edge middleware should encode their state into the JWT claim, not require a DB read; allowlists for path-bypass MUST be exact-match Sets, never prefix-match.
